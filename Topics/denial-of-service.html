Denial of Service (DoS/DDoS)

Overview

A Denial of Service (DoS) attack aims to make a machine or network resource unavailable to its intended users. This is achieved by flooding the target with traffic or sending information that triggers a crash. In 2025, these attacks frequently utilize AI-driven botnets to bypass traditional rate-limiting controls.

Attack Description

Adversaries use DoS/DDoS for extortion, distraction during a separate breach, or ideological reasons. Attacks generally fall into three categories:

  • Volumetric: Saturates bandwidth with massive amounts of traffic.
  • Protocol: Exploits weaknesses in Layer 3 or 4 (e.g., SYN Floods).
  • Application Layer: Targets specific web functions (e.g., heavy database queries).

Investigation Workflow

Characterize the Attack

Type of DoS:

  • Network Flood (SYN, UDP, ICMP): Volume-based exhaustion of network pipes.
  • Application Layer (HTTP Flood, Slowloris): Focused resource exhaustion on web servers.
  • Distributed (DDoS): Coordinated attack from multiple global sources.
  • Amplification Attack: Reflected traffic (e.g., DNS, NTP, or Memcached) to multiply volume.

Volume and Severity:

  • Quantify traffic volume (Gbps) and throughput (Packets Per Second).
  • Identify the number of unique attack sources/IPs.
  • Note the duration of the impact and which specific services are failing.

Identify Target and Impact

What is being targeted?

  • Specific application or API endpoint.
  • Entire network segment or CIDR block.
  • Individual server IP address.
  • DNS infrastructure (which can take down all associated services).

Current and Business Impact:

  • Service Status: Is it completely down, degraded, or partially out?
  • Operations: Are customer-facing services or internal employee operations disrupted?
  • Risk: Assess immediate financial loss and long-term reputation damage.

Identify Attack Source(s)

Source IP Analysis:

  • Single Source: Block immediately at the local firewall.
  • Multiple Sources (DDoS): Coordinate with ISP or scrubbing service.
  • Spoofed Sources: Common in amplification; local blocking is often ineffective.
  • Geolocation: Identify patterns that might suggest a regional campaign.

Attack Characteristics:

  • Legitimate-Looking: Likely an Application Layer attack targeting software logic.
  • Malformed Packets: Network Layer attack targeting hardware/OS stacks.
  • Botnet Signatures: Cross-reference sources with known botnet families (e.g., Mirai variants).

Implement Immediate Mitigation

Network Level:

  • Block attacking IPs at the perimeter firewall.
  • Implement rate limiting on the edge router.
  • Enable hardware anti-DDoS features.
  • Contact ISP for Upstream Blocking or Remote Triggered Black Hole (RTBH).
  • Route traffic to a specialized scrubbing center (e.g., Cloudflare, Akamai).

Application Level:

  • Enable/Force CDN caching.
  • Implement CAPTCHA challenges to filter bot traffic.
  • Adjust Web Application Firewall (WAF) rules to drop suspicious headers.
  • Scale cloud resources (auto-scaling) if budget permits.
  • Activate geo-blocking if the attack is localized to specific regions.

Is Mitigation Effective?

  • Yes:
    • Continue enhanced monitoring.
    • Document the attack timeline and effectiveness of each control.
    • Maintain heightened alertness for secondary "distraction" attacks.
  • No:
    • Escalate to Senior Analysts and Network Architecture teams.
    • Engage specialized DDoS mitigation vendors.
    • Consider taking the service offline temporarily to protect backend infrastructure.

Tools & Queries

  • NetFlow Analysis (Top Talkers by Source/Destination IP)
  • WAF Logs (Check for high request rates per Session/IP)
  • Firewall Throughput Monitors
  • SNMP monitoring for CPU/Memory on network appliances
  • External Monitoring (e.g., Pingdom, ThousandEyes) for uptime status

Response Workflow

  • Identify the attack type and implement the appropriate mitigation layer.
  • Coordinate with ISPs to handle volumetric floods before they reach the pipe.
  • Document the attack and update the DoS response playbook based on findings.
  • Conduct a post-mortem to identify vulnerabilities or missing security controls.
